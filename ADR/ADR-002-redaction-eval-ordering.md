# ADR-002: Порядок redaction и eval-гейта — обратимая токенизация, grounding по provenance

- **Статус:** Accepted
- **Дата:** 2026-06-13
- **Контекст:** [BRIEF.md](../BRIEF.md), противоречие C3

## Контекст

Два constraint'а брифа конфликтуют по порядку слоёв:

1. **Redaction** — маскировать PII/секреты **до отправки в модель** (banking-
   sensitive data classification, Sber-периметр).
2. **Eval-гейт доверия** — каждый LLM-вывод проходит самопроверку **против
   исходных данных**.

Если в модель уходит маскированный текст, против чего сверяет гейт? Против
маскированного — не ловит искажения по содержанию; против сырого — сам гейт
(LLM-вызов) нарушает периметр.

## Решение

**Redaction = обратимая токенизация** (не разрушающая маскировка). **Eval-гейт
работает на masked-vs-masked и проверяет provenance/структуру, а не контент
секрета.**

```
tracker-MCP ─raw─▶ [Redaction proxy] ─masked─▶ LLM ─masked out─▶ [Eval-gate] ─pass─▶ [Re-hydrate] ─▶ показ/запись
                         │                                                                  ▲
                    map{token→real}  ── client-side, в LLM не уходит ─────────────────────────┘
```

1. Классифицированные поля → **стабильные плейсхолдеры** `⟦PII:7a3⟧`; карта
   `token→real` живёт только client-side, в модель не уходит.
2. LLM получает masked-вход, отдаёт masked-выход (плейсхолдеры протекают насквозь).
3. **Eval-гейт сверяет masked-выход с masked-источником**, проверяя **provenance**:
   каждый id/число/ссылка/токен в выводе трассируется в источник; нет выдуманных
   id; счётчики сходятся.
4. **После** прохождения гейта — client-side re-hydration токенов в реальные
   значения, внутри периметра.

**Почему снимает противоречие:** grounding есть проверка *происхождения и
структуры*, а не чтение содержимого секрета — поэтому гейту сырой PII не нужен.
Неклассифицированный контент (заголовок «fix payment timeout») **не** токенизи-
руется → гейт видит его в открытую и проверяет контентную верность как обычно.
Итог: полная точность гейта по несекретному контенту + структурная по секретному,
ни один сырой PII не покидает периметр.

## Trade-off / требования

- **Детерминизм:** один секрет → один токен в рамках сессии, иначе модель не
  свяжет «того же человека» между карточками.
- **Карта токенов** — session-scoped state; её нельзя логировать в transcript
  модели.
- **Остаточный риск** переезжает с «гейт слепнет» на «redaction пропустил PII» →
  тест на утечку сырого PII (gate-тест из брифа) остаётся обязательным.

## Последствия

- `references/redaction-rules.md` определяет классы полей и правила токенизации.
- Eval-гейт специфицируется как provenance-checker над masked-парой, не как
  контентный сравниватель.
- Что пересмотреть: если появятся выводы, требующие семантики секрета (напр.
  дедуп людей по содержимому) — токенизация должна нести типизированные,
  сравнимые токены, а не плоские.
