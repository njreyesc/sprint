# ADR-004: Граница скилла — планирование vs ведение спринта

- **Статус:** Accepted
- **Дата:** 2026-06-13
- **Деки:** владелец продукта teamlead-copilot
- **Контекст:** [BRIEF.md](../BRIEF.md), [ADR-001](ADR-001-sprint-memory.md)

## Контекст

teamlead-copilot спроектирован как **один** скилл с режимами поверх трекера:
Daily-сводка, create/search задач, связи/блокеры, планирование спринта, итоги
спринта (план vs факт → вход следующего планирования). Возник вопрос: разделить
его на два скилла —

- **«Ведение спринта»** — высокочастотная рутина внутри активного спринта:
  Daily, create, search, блокер-граф;
- **«Планирование»** — эпизодическая работа на стыке спринтов: итоги (retro) +
  раскладка следующего состава по velocity.

Силы, действующие на решение:

- **Разный такт и поверхность запуска.** Ведение — ежедневно (scheduled-сводка +
  интерактив), планирование — раз в ~2 недели, интерактивно. Бриф уже разводит
  поверхности, но *внутри* ведения (Daily = scheduled), а не по линии
  планирование/ведение.
- **Разный «вес» данных.** Ведение читает активный спринт. Планирование тянет
  историю закрытых спринтов и velocity report (greenhopper-эндпоинты, [ADR-001](ADR-001-sprint-memory.md)) —
  это отдельная, полу-приватная механика.
- **Общий слой доверия — существенный.** Redaction, eval-гейт, grounding,
  подтверждение-перед-записью, provenance, измерение свежести данных — общие для
  *всех* режимов. Gate-метрика «ноль выдуманных фактов» — экзистенциальная.
- **Итоги спринта — шарнир.** Retro одновременно конец ведения (план vs факт
  активного спринта) и вход планирования. Линия раздела проходит ровно через
  самый связанный кусок.
- **Композитные запросы.** Бриф явно держит композитный тест «дейли и
  заведи задачи по блокерам» — роутер поверх режимов в одном контексте.
- **Открытое решение по среде исполнения** (custom loop vs сам Claude) меняет
  цену границы: для Claude-скилла размер SKILL.md и чёткость роутера важны
  сильнее, чем для оркестратора с внешним роутингом.

## Решение

**Не делить сейчас.** Оставить один скилл с раздельными режимами внутри, но
**с первого дня вынести общий слой доверия в `references/`** как разделяемый
контракт, чтобы будущий split был дешёвым и обратимым.

Раздел проводится не «политически», а по факту общей подложки: redaction-rules,
eval-гейт, grounding-инварианты, tracker-mapping, state=`derived-on-read` живут в
`references/` и не дублируются. Режимы — единицы внутри одного SKILL.md под одним
роутером.

## Рассмотренные опции

### Опция A: Один скилл, общий слой в `references/` (выбрано)

| Измерение | Оценка |
|-----------|--------|
| Сложность | Низкая — текущая структура, режимы уже разделены |
| Риск дрейфа safety-контракта | Низкий — один источник правды по доверию |
| Композитные запросы | Нативно — один роутер, один контекст |
| Цена будущего split | Низкая — подложка уже вынесена |

**Плюсы:** единый экзистенциальный слой доверия без дублей; шарнир-retro не
режется; композитные сценарии работают; YAGNI + обратимость.
**Минусы:** SKILL.md растёт; в контексте ежедневного Daily лежат и инструкции
планирования (лишние токены, шум для роутера).

### Опция B: Два скилла — «ведение» и «планирование»

| Измерение | Оценка |
|-----------|--------|
| Сложность | Средняя — два SKILL.md, общий слой надо извлекать всё равно |
| Риск дрейфа safety-контракта | Высокий, если подложку не вынести; иначе как в A |
| Композитные запросы | Ломаются на границе скиллов / нужен внешний роутер |
| Фокус контекста | Лучше — каждый скилл грузит только своё |

**Плюсы:** меньше контекст на запуск; независимый eval (daily-grounding vs
velocity-provenance); независимая эволюция; чистый такт-раздел.
**Минусы:** retro-шарнир рвётся по живому; риск двух копий gate-машинерии;
кросс-скилловые композиты; преждевременно при открытой среде исполнения.

## Trade-off

Ключевой компромисс — **фокус контекста (за split) против целостности слоя
доверия и retro-шарнира (за единство)**. Поскольку gate-метрика
экзистенциальна, цена дрейфа safety-контракта между двумя копиями выше, чем
выигрыш от более лёгкого контекста. Выигрыш фокуса при этом **достижим и без
split** — через вынос подложки в `references/` и ленивую подгрузку тяжёлой
velocity-механики только в режиме планирования.

Раздел B остаётся правильным ходом *позже*: его делает дешёвым именно вынос
подложки, сделанный в A. То есть A — не «против» B, а его подготовка.

## Триггеры пересмотра (когда резать на два)

Делить, как только сработает хотя бы один:

1. **Деградация роутинга/инструкций** в едином SKILL.md (промахи режима,
   «протекание» планирования в ежедневные ответы).
2. **Velocity-механика планирования** (greenhopper, история, [ADR-001](ADR-001-sprint-memory.md))
   заметно утяжеляет ежедневный путь даже при ленивой подгрузке.
3. **Среда исполнения зафиксирована как Claude-скилл** и размер/такт двух частей
   расходятся настолько, что нужен раздельный жизненный цикл и владение.
4. **Расходятся eval-гейты и поверхности запуска** так, что общий релиз-цикл
   мешает (планирование релизится реже/осторожнее ведения).

## Последствия

- **Что становится проще:** один слой доверия → одна точка для redaction/eval/
  grounding; композитные запросы; обратимость решения.
- **Что становится сложнее:** держать SKILL.md компактным; нужна дисциплина
  ленивой подгрузки тяжёлых references в планировании.
- **Что вынести в `references/` сейчас (чтобы split был дешёвым):**
  redaction-rules, eval-гейт/grounding-инварианты, tracker-mapping (вкл. контракт
  «velocity-история»), state-модель `derived-on-read`.
- **Шов для будущего split (если сработает триггер):** по такту —
  *ведение* {Daily, search, create, связи} и *планирование* {retro, раскладка};
  retro уходит в планирование как его prep-фаза, оба потребляют velocity-историю.
- **Что пересмотреть:** при фиксации среды исполнения (открытое решение из
  брифа) перечитать триггер 3.
