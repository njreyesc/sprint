# ADR-001: Память спринтов — нативная история Jira, derived-on-read без агентского стора

- **Статус:** Accepted (заменяет ранний вариант с агентским snapshot-логом)
- **Дата:** 2026-06-13
- **Контекст:** [BRIEF.md](../BRIEF.md), противоречие C2

## Контекст

Бриф содержит инвариант «источник правды — трекер; агент ничего не хранит у себя
как истину» и одновременно требует режим «память спринтов»: итоги спринта
(plan vs факт, velocity) используются как вход планирования следующего. Ранний
вариант решения вводил агентский append-only снэпшот закрытых спринтов — это
второй стор, который может дрейфовать от трекера.

**Решение по выбору трекера: целевой трекер — Jira.** Это снимает само основание
противоречия: история спринтов в Jira хранится нативно, отдельный стор не нужен.

## Решение

**`derived-on-read` распространяется и на историю — источником служит нативная
история спринтов Jira. Агент не персистит ничего.**

- **Текущий прогресс / velocity** — пересчёт из активного спринта Jira на каждый
  запрос.
- **Историческая velocity** — читается из закрытых спринтов Jira:
  - `GET /rest/agile/1.0/board/{boardId}/sprint?state=closed` — список закрытых
    спринтов (с `completeDate`, `startDate`, `endDate`, `goal`).
  - velocity report по доске: `GET /rest/greenhopper/1.0/rapid/charts/velocity?rapidViewId={boardId}`
    → `velocityStatEntries[sprintId] = { estimated, completed }` (committed vs
    выполнено по каждому спринту).
  - при необходимости детализации — `GET /rest/greenhopper/1.0/rapid/charts/sprintreport?rapidViewId={boardId}&sprintId={id}`
    (completedIssues, puntedIssues и т.п.).

**Immutable-факт «как было на закрытии» владеет Jira:** `completeDate` и
committed/completed по спринту фиксируются при закрытии спринта. Агенту незачем
их копировать.

Инвариант «агент ничего не хранит как истину» держится **в полной силе** — даже
история живёт в трекере. Планирование по-прежнему помечает основу: «по N закрытым
спринтам Jira» либо «нет истории (cold-start)» — последнее не подставляет
оптимистичную оценку (tension T5).

## Trade-off

- **Привязка к Jira.** Решение сознательно сужает трекер-агностичность (бриф,
  T3): для трекеров без нативной истории спринтов вернётся агентский
  snapshot-фолбэк из раннего варианта. «Velocity-история» выносится в
  `references/tracker-mapping.md` как **контракт**, который Jira закрывает
  нативно, а другие трекеры — фолбэком.
- **Полу-приватный API.** velocity/sprintreport — это greenhopper-эндпоинты
  (не стабильный публичный Agile REST). Коупинг-риск изолируется за контрактом
  «velocity-история» в маппинге.
- **Правки после закрытия.** Если SP закрытого спринта отредактируют в Jira,
  velocity report может пересчитаться. Принимаем: Jira — источник правды, её
  представление истории и есть честная база; агент не «чинит» это копией.

## Последствия

- State-модель скилла **не** содержит стора истории спринтов (минус компонент
  против раннего варианта).
- `references/tracker-mapping.md` определяет контракт «velocity-история» с
  Jira-реализацией на Agile + greenhopper API.
- Mock tracker-mcp эмулирует Jira-семантику: закрытые спринты с `completeDate` и
  committed/completed, чтобы прогонять планирование на истории без боевой Jira.
- Любой вывод про velocity несёт provenance: `jira-closed-sprint(N)` |
  `active-sprint` | `cold-start`.
- Что пересмотреть: при подключении не-Jira трекера — активировать
  snapshot-фолбэк; при большой истории — окно/затухание velocity.
