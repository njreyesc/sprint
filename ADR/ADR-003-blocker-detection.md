# ADR-003: Детекция блокеров — три класса, базовый контур на activity-дельтах

- **Статус:** Accepted
- **Дата:** 2026-06-13
- **Контекст:** [BRIEF.md](../BRIEF.md), противоречие C1

## Контекст

Метрика успеха обещала «блокеры всплывают в день возникновения». Constraint'ы
брифа этому противоречат: данные в трекере протухшие (статусы не двигают), а
коннекторы (Slack/Calendar) объявлены «бонусом, не базой». На одном трекере со
стухшими статусами блокер, который разработчик не занёс, день-в-день не виден —
метрика опиралась на свежесть, которую сам бриф объявляет недостижимой.

## Решение

**Расщепить «блокер» на три класса и привязать метрику к достижимости на
базовом контуре.**

| Класс | Сигнал | Контур |
|---|---|---|
| **Явный блокер** | связь `blocked by` / статус `Blocked` | базовый (трекер) |
| **Застой (stall)** | задача `in progress` N дней без изменения статуса/комментария/активности | базовый (трекер) |
| **Немой блокер** | завис, следа нет ни в задаче, ни в активности | вне базы → опц. Slack |

Ключ: **stall выводится из activity-timestamps трекера** (тот же `since`-дельта-
механизм из открытых решений брифа), а не из коннектора. Он делает двойную
работу — сигнал блокера и измеритель свежести данных.

**Переформулировка метрики** (внесена в BRIEF.md): «Явные блокеры и застой
подсвечиваются в день, когда отражены в трекере или в *отсутствии* активности по
задаче. Немой блокер, не оставивший следа, — вне гарантии базового контура».

## Trade-off

- **Stall-порог N — эвристика:** даёт false-positive на легитимно-долгих задачах.
- **Принимаем:** подсветка ≠ обвинение, тимлид фильтрует; дешевле пропуска.
- Slack теперь честно закрывает **только** немой блокер, а не всю метрику.

## Последствия

- Базовый контур читает activity-timestamps трекера → mock tracker-mcp обязан
  отдавать историю активности/изменений, не только текущее состояние.
- Stall-порог N — конфигурируемый параметр (дефолт уточнить замером).
- Связка: stall-сигнал переиспользуется индикатором свежести данных (см. раздел
  «Свежесть данных» в BRIEF.md).
- Что пересмотреть: per-команда/per-тип калибровка N вместо единого порога.
