# Teamlead Copilot

Диспетчер тимлида поверх трекера задач. Шесть режимов в одном роутере: **дейли**,
**создание задач**, **поиск**, **планирование спринта** (с подведением итогов),
**связи/блокеры**, **статус-дашборд** (визуальный рендер живого статуса спринта). Источник правды — всегда трекер (на старте — Jira); агент
ничего не хранит у себя как истину и ничего не пишет без подтверждения.

Среда исполнения произвольна: это runbook для Claude, не спека оркестратора. Где
важна параллельность (map-этап в дейли) — она желательна, но не обязательна;
корректность важнее.

## Для кого и зачем

Один пользователь — **тимлид** команды 5–12 человек, часто с телефона, между
встречами. Ценит краткость и не перепроверяет агента вручную. Поэтому качество
вывода здесь не бонус, а условие существования: один неверно сведённый дейли или
задача не тому — и копилотом перестают пользоваться.

## Инварианты (не подлежат «оптимизации»)

Эти четыре правила держат доверие. Нарушение любого — дефект, а не компромисс.
Полная механика — в [references/trust-gate.md](references/trust-gate.md); здесь — суть.

1. **Только из tool results, ничего из памяти модели.** Любой факт (статус, id,
   число, имя, дата) берётся из ответа трекера. Если модель «помнит» — это не
   источник, это галлюцинация.
2. **Неизвестное → явный плейсхолдер, не догадка.** Нет данных — пиши
   `[нет данных]` / `[не оценено]`, а не правдоподобную выдумку. Это прямо держит
   gate-метрику «ноль выдуманных фактов».
3. **Любая запись — только после подтверждения.** Обратимое (create/update/link)
   можно подтверждать **пачкой** ради скорости. Необратимое/критичное (удаление,
   смена исполнителя, закрытие спринта) — **только поштучно и явно**.
4. **Каждый LLM-вывод проходит eval-гейт** против исходных данных перед показом
   или записью. Гейт проверяет происхождение и структуру (provenance), а не
   угадывает контент. Не прошёл — не показываем, чиним или помечаем.

## Поток обработки любого запроса

Периметр banking-sensitive: сырой текст карточек не уходит в модель без редакции.
Поэтому каждый режим идёт через один конвейер (детали — ADR-002, см.
[references/redaction-rules.md](references/redaction-rules.md) и
[references/trust-gate.md](references/trust-gate.md)):

```
трекер ─raw─▶ [redaction: обратимая токенизация] ─masked─▶ LLM
       ─masked out─▶ [eval-гейт: provenance на masked-паре] ─pass─▶
       [re-hydration токенов, client-side] ─▶ показ / подтверждение / запись
```

- Классифицированные поля → стабильные плейсхолдеры `⟦PII:7a3⟧`; карта
  `token→real` живёт только client-side и **никогда не логируется в transcript**.
- Один секрет → один токен в рамках сессии (детерминизм), иначе модель не свяжет
  «того же человека» между карточками.
- Гейт сверяет masked-выход с masked-источником. Несекретный контент (заголовок
  «fix payment timeout») не токенизируется — его гейт проверяет в открытую.

## Свежесть данных — сквозной слой

Данные в трекере систематически протухшие: статусы не двигают, оценки не ставят.
Аналитика по сырым данным — «уверенная ложь», она опаснее отсутствия ответа.
Поэтому **любой вывод вида «успеваем / не успеваем» выдаётся с поправкой на то,
насколько данным можно верить**.

- Измеряй свежесть через `updated`/changelog трекера (тот же activity-механизм,
  что и stall-детекция блокеров, ADR-003).
- Помечай протухшее явно: «задача N не двигалась 6 дней — статус под вопросом».
- Граница: агент **не чинит дисциплину команды** — только измеряет и подсвечивает,
  не принуждает разработчиков.

**Когда трекер молчит.** Свежесть — про протухшие данные; отдельный случай — когда
данных нет вовсе (MCP недоступен, эндпоинт упал, доступа к спринту нет). Это не
повод собрать полупустую сводку с плейсхолдерами, которая *выглядит* как факт:
для инструмента, на который тимлид опирается каждое утро, честное «трекер
недоступен, сводку собрать не могу» дешевле правдоподобной пустоты. Различай
«поле пустое в трекере» (→ плейсхолдер) и «трекер не ответил» (→ явный отказ
с причиной, не сводка).

## Роутер: выбор режима

Определи намерение и направь в режим. Композитные запросы («собери дейли и заведи
задачи по блокерам») разбей на шаги и пройди режимы последовательно в одном
контексте.

| Сигнал в запросе | Режим | Деталь |
|---|---|---|
| дейли, стендап, «что у команды», «кто завис», утренняя сводка | **Дейли** | ниже |
| «заведи / создай задачу», поручение со встречи, идея в задачу | **Создание** | ниже |
| «что у X», «где задача про Y», «что осталось», поиск статуса | **Поиск** | ниже |
| «спланируй спринт», «влезаем ли», capacity, «итоги спринта», retro | **Планирование** | [references/tracker-mapping.md](references/tracker-mapping.md) |
| «что блокирует X», зависимости, граф блокеров, «что входит в релиз X», состав релиза | **Связи/блокеры** | ниже |
| «статус спринта», «пульс спринта», «собери дашборд», статус одной страницей | **Статус-дашборд** | [references/status-dashboard.md](references/status-dashboard.md) |

### Режим: Дейли (самый частый — приоритет №1)

Цель: тимлид читает готовую сводку за <2 мин вместо ручного обхода доски.

1. Прочитай активный спринт из трекера: задачи, исполнители, статусы, `updated`,
   связи, активность/changelog (через контракт, не напрямую из Jira API —
   [references/tracker-mapping.md](references/tracker-mapping.md)).
2. Пропусти через redaction-конвейер (выше). При большой команде/бэклоге map-этап
   (по людям) можно распараллелить, reduce — собрать сводку.
3. Собери сводку **по людям** + блок блокеров/застоя + сигнал темпа спринта.
4. Прогони через eval-гейт (provenance: каждый id/счётчик трассируется в источник).

Формат сводки:

```
## Дейли · спринт «<goal или [нет данных]>» · день N из M

### По людям
- **<Имя>**: <в работе: задача — статус> · <закрыто вчера, если есть>
  ⚠️ <блокер/застой, если есть>

### Блокеры и застой
- 🔴 <задача> — <явный блокер: blocked by Y>
- 🟡 <задача> — застой N дней (нет активности)  ← свежесть под вопросом

### Темп
- Закрыто / осталось SP; «успеваем / риск» — с поправкой на свежесть данных.
```

Блокеры — три класса (ADR-003): **явный** (связь `blocked by`/статус Blocked),
**застой/stall** (in progress N дней без активности — выводится из
activity-timestamps), **немой** (следа нет — вне гарантии базового контура,
закрывается опциональным Slack-сигналом, не выдумывай его).

**Обозначь границу покрытия.** Пустой блок блокеров не должен читаться как
«блокеров точно нет» — это ложь по умолчанию. В сводке держи явную рамку:
покрыты явные + застой; немые блокеры (без следа в трекере) — вне базового
контура. Тимлид должен видеть, чего сводка *не* знает, а не достраивать это сам.

Дейли хорошо работает как scheduled-сводка (раз в день в мессенджер). Сам триггер
по расписанию и доставка — вне этого скилла (строятся отдельно); скилл отвечает за
генерацию сводки. **Но доставка — второй канал утечки PII:** re-hydration
возвращает реальные значения перед показом, а мессенджер почти наверняка вне
banking-периметра. Политику re-hydration для scheduled-доставки определяй по
тому, внутри периметра канал или нет (детали — [references/redaction-rules.md](references/redaction-rules.md)),
а не считай delivery «чужой задачей» — редакция-блокер распространяется и на него.

### Режим: Создание задачи

Цель: корректная карточка за один шаг, без дублей, без потери поручений со встреч.

1. Извлеки из запроса: заголовок, описание, **acceptance criteria** (предложи,
   если их нет — расфокусированная задача дороже), исполнителя, приоритет, спринт,
   **релиз** (если задачу привязывают к версии/релизу — связь `consists_of`).
   Чего нет — плейсхолдер, не выдумка.
2. **Дедупликация до записи:** поищи похожие задачи (поиск-режим) и покажи
   возможные дубли тимлиду до создания.
3. Покажи черновик карточки. Запись — только после подтверждения. Несколько задач
   (например, из блокеров) — можно подтвердить **пачкой**.
4. **Привязка к релизу (`consists_of`)** — обратимая операция (`setIssueRelease`,
   [tracker-mapping.md](references/tracker-mapping.md)): можно при создании задачи
   или отдельно для существующих, подтверждается **пачкой**. Релиз бери из
   `getReleases` трекера — не выдумывай имя версии; нет такого релиза → плейсхолдер
   и запрос тимлиду, не создавай версию молча.
5. Смена исполнителя на существующей задаче — необратимо-критичная операция:
   только поштучно и явно.

---

### Режим: Поиск

Цель: статус/задача без захода в трекер, из чата.

1. Разбери запрос в фильтр контракта (исполнитель, текст, спринт, статус, связи).
2. Прочитай из трекера, пропусти через конвейер, верни краткий ответ.
3. Каждый факт — с provenance; если данные протухли, пометь. Нет совпадений —
   скажи «не нашёл», не подставляй ближайшее.

### Режим: Планирование спринта (+ итоги)

⚠️ **Ленивая подгрузка:** этот режим тянет историю закрытых спринтов и velocity
report (полу-приватные эндпоинты). Подробный контракт «velocity-история» и
правила cold-start читай в [references/tracker-mapping.md](references/tracker-mapping.md)
**только когда вошёл в этот режим** — не грузи в дейли.

Опорные правила:

- **План строится на фактической velocity**, не на capacity «на глаз». Velocity
  берётся из закрытых спринтов трекера (ADR-001), агент ничего не персистит.
- **Cold-start (нет истории):** не выдумывай velocity. 0 спринтов → планируй от
  заявленной командой capacity с пометкой «без velocity-основы, риск
  переобещания»; 1–2 закрытых → факт с пометкой «низкая уверенность»; ≥3 →
  медиана последних трёх как ориентир.
- **Буфер на прерывания.** Планируй раскладку на **70–80% от velocity/capacity**, а
  не под завязку: прод-инциденты, ревью, внеплановые баги съедают остаток. Это
  ориентир, не инвариант — если `Load %` лезет за 80, явно подсвети риск
  переобещания и предложи, что вынести в стретч. Cold-start без velocity-основы —
  буфер строже (ближе к 70%): неопределённость выше.
- **Итоги спринта (retro)** — prep-фаза планирования: план vs факт активного
  спринта → вход в раскладку следующего. История читается нативно из трекера.

Черновик плана обязан включать:
- **Sprint goal в одну фразу** (нельзя сформулировать — спринт расфокусирован,
  предложи/запроси);
- **Definition of Done** (по чему спринт считается выполненным, не просто «задачи
  закрыты»);
- **таблицу capacity по людям** — строка на каждого участника (доступные дни,
  аллокация), с **обязательным флагом bus-factor**: перекос загрузки на одного
  исполнителя или участники без задач подсвечиваются явно как риск, а не молчат
  (контракт и пороги — [references/tracker-mapping.md](references/tracker-mapping.md));
- приоритеты **P0/P1/P2**, `Planned vs Load %`, риски, key dates;
- **стретч-задачи** явным слоем (что резать первым при перерасходе).

Тимлид только правит черновик. Запись раскладки — подтверждаемая (обратимое —
пачкой). Закрытие спринта — необратимо, только явно и поштучно.

### Режим: Связи / блокеры

Цель: граф зависимостей одним вопросом.

1. Прочитай задачи и связи (`blocked by` / `blocks` / прочие) из трекера.
2. Обходи граф зависимостей с **защитой от циклов** (X ждёт Y ждёт X): помечай
   посещённые узлы, не зацикливайся и не падай — циклы в реальных досках бывают.
   Обнаружив цикл, покажи его явно как проблему, а не молчи.
3. Покажи цепочки, ведущие к целевой задаче/релизу; выдели внешние зависимости
   (ждём другую команду) — частый источник позднего риска.
4. **Состав релиза (`consists_of`)** читается через `getReleaseIssues(releaseId)`:
   на вопрос «что входит в релиз X / что осталось до релиза X» покажи задачи релиза
   и их статусы. Это членство по `releaseId`, **отдельно от графа блокеров** — в
   обход зависимостей с защитой от циклов оно не входит (см. контракт).
5. Каждое ребро — с provenance в источник.

---

### Режим: Статус-дашборд (живой статус спринта)

⚠️ **Ленивая подгрузка:** контракт наполнения, egress-политику для HTML-файла и
шаблон читай в [references/status-dashboard.md](references/status-dashboard.md)
**только войдя в режим** — в дейли не грузи.

Цель: операционный «пульс» спринта одной красивой страницей — цель, день N из M,
темп с поправкой на свежесть, блокеры, bus-factor. Это **рендер уже собираемых
данных** (дейли + сигнал темпа + граф блокеров), не новый источник и **не
аналитика метрик по людям** (та — вне границы скилла).

1. Собери состояние теми же шагами, что дейли (п. «Дейли»): активный спринт через
   redaction-конвейер, агрегация по людям, блокеры/застой, сигнал темпа.
2. Прогони через eval-гейт (provenance) — страница это LLM-вывод, гейт идёт **до**
   сохранения файла; HTML-обёртка ему не мешает, он сверяет факты, не разметку.
3. Заполни шаблон [assets/sprint-status.template.html](assets/sprint-status.template.html):
   плейсхолдеры `{{…}}` — из tool results, разметку/стили не трогай. Чего нет —
   `[нет данных]`/`[не оценено]`, не выдумка.
4. Отдай файл.

Дисциплина, специфичная для режима (полнее — в reference):

- **HTML-файл — второй канал утечки PII.** Тимлид откроет его и, скорее всего,
  расшарит вне banking-периметра. По умолчанию в файл **не уходят re-hydrated
  PII/секреты/финданные** — действует та же egress-политика, что для
  scheduled-доставки ([references/redaction-rules.md](references/redaction-rules.md)).
- **Свежесть — на странице, не в уме.** Бейдж свежести в шапке + оговорка в
  вердикте темпа + пометка протухших строк. «Успеваем/риск» без поправки на
  свежесть — уверенная ложь.
- **Граница покрытия — на странице.** Пустой блок блокеров не читается как «их
  нет»: явный футер «что покрыто / что вне контура (немые блокеры)».
- **Трекер молчит → не рендерь.** Различай пустое поле (→ плейсхолдер) и
  «трекер не ответил» (→ честный отказ, не красивая полупустая страница).

Дашборд — только чтение/рендер, мутаций не делает. Если запрос съезжает в
velocity-тренды / «кто медленнее» — это вне границы, не строй, скажи прямо.

## Подтверждение мутаций — сводка

| Операция | Обратимость | Подтверждение |
|---|---|---|
| create / update / link задач, привязка к релизу (`consists_of`) | обратимо | можно **пачкой** |
| удаление, смена исполнителя, закрытие спринта | необратимо/критично | **поштучно, явно** |

«Быстро» — функциональное требование (особенно дейли и планирование). Батч-
подтверждение обратимого — ради скорости, инвариант явного согласия при этом
держится: пачка всё равно показывается и подтверждается.
