# Status-dashboard — живой статус спринта (визуальный рендер)

Режим **Статус-дашборд** собирает операционный «пульс» активного спринта на одной
красивой HTML-странице: цель, день N из M, темп с поправкой на свежесть, блокеры,
bus-factor. Это **рендер того, что скилл уже производит** (дейли + сигнал темпа +
граф блокеров), а не новый источник данных и **не аналитика метрик по людям** —
последнее остаётся вне границы скилла (см. `../SKILL.md`, список «НЕ для»).

Шаблон: [../assets/sprint-status.template.html](../assets/sprint-status.template.html) —
самодостаточный (инлайн-CSS, дизайн-система Anthropic-палитры, как в
`../evals/eval-viewer/viewer.html`). Агент заполняет плейсхолдеры `{{…}}`,
не меняя разметку и стили.

## Граница: почему это не нарушает «НЕ для дашбордов»

Скилл исключает **аналитику/метрики по людям** (burndown-тренды как метрика,
velocity-чарты, нагрузка как KPI — близко к перформанс-ревью). Статус-дашборд
исключению не противоречит, потому что:

- рендерит **текущее состояние**, а не тренды/историю как метрику;
- те же поля, что уже отдаёт дейли-режим (задача, статус, блокер, застой);
- bus-factor и простой подаются как **операционный риск спринта**, а не как
  оценка человека.

Если запрос съезжает в «покажи graph velocity за полгода / кто медленнее всех» —
это уже вне границы: не строй, скажи прямо.

## Инвариант egress (главный риск этого режима)

HTML-файл — **второй канал утечки PII**, ровно как scheduled-доставка в мессенджер
([redaction-rules.md](redaction-rules.md) §«Второй канал утечки»). Тимлид откроет
его в браузере и, скорее всего, кинет в чат/шару — это почти наверняка **вне
banking-периметра**.

- В файл **не уходят re-hydrated PII/секреты/финданные.** Re-hydration реальных
  значений — только если рендер заведомо остаётся внутри периметра.
- По умолчанию (файл наружу): состав — нечувствительные поля (заголовки задач,
  статусы, счётчики, id, имена команды, если `assignee` не классифицирован как
  PII). Клиентские данные из описаний — не вытаскивать.
- Тот же gate-тест на утечку сырого PII распространяется на payload дашборда, а не
  только на LLM-payload.

## Что наполняет страницу (всё — из tool results)

| Блок | Источник | Правило |
|---|---|---|
| Цель спринта, день N/M | активный спринт трекера | нет цели → `[нет данных]`, спринт расфокусирован |
| Темп (SP закрыто/осталось/план, %) | задачи + оценки | оценок нет → `[не оценено]`, не выдумывать |
| Вердикт темпа | производный | **никогда без оговорки на свежесть** (см. ниже) |
| По людям | дейли-агрегация по `assignee` | задача · статус · закрыто вчера |
| Bus-factor / простой | контракт capacity ([tracker-mapping.md](tracker-mapping.md)) | перекос на одного / участник без задач → явный флаг |
| Блокеры, застой | граф связей + activity-timestamps (ADR-003) | классы: явный, застой |
| Граница покрытия | константа режима | что покрыто + что вне контура (немые блокеры) |

## Свежесть — на странице, а не в уме

Данные систематически протухшие; «успеваем/не успеваем» без поправки — уверенная
ложь ([trust-gate.md](trust-gate.md)).

- **Бейдж свежести** в шапке (`ok` / `warn` / `stale`) — по доле задач, чей
  `updated` старше порога застоя. Не косметика: задаёт доверие ко всей странице.
- Вердикт темпа всегда несёт оговорку (`{{TEMPO_CAVEAT}}`): «при условии, что
  статусы актуальны; N задач не двигались >K дней».
- Протухшие позиции помечаются «свежесть под вопросом» прямо в строке.

## Когда трекер молчит

Различай как везде: **поле пустое** → плейсхолдер в ячейке; **трекер не ответил**
(MCP недоступен, нет доступа к спринту) → **не рендерь** полупустую красивую
страницу, которая *выглядит* как факт. Честный отказ с причиной дешевле
правдоподобной пустоты.

## Eval-гейт

Страница — LLM-вывод: проходит provenance-гейт **до** сохранения файла, как любой
другой вывод. Каждый id/счётчик трассируется в источник; не прошёл — не отдаём
файл, чиним или помечаем. HTML-обёртка гейту не мешает — он сверяет факты, не
разметку.
