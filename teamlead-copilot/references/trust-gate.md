# Trust gate — eval-гейт, grounding, свежесть

Экзистенциальный слой: тимлид не перепроверяет агента вручную, поэтому eval-гейт —
единая точка отказа и должен держать gate-метрику **«ноль выдуманных фактов»**
(BRIEF.md). Это functional requirement, не опция. Общий для всех режимов (ADR-004).

## Что проверяет гейт

Гейт работает на **masked-паре** (masked-выход модели vs masked-источник из
трекера) и проверяет **происхождение и структуру (provenance)**, а не контент
секрета — поэтому ему не нужен сырой PII (ADR-002).

Проверки:

1. **Трассируемость.** Каждый `id` / число / ссылка / токен в выводе
   прослеживается в источник. Есть в выводе, нет в источнике → выдумка → блок.
2. **Нет выдуманных id.** Любой `issueId`/`sprintId`/токен — только из tool result.
3. **Сходимость счётчиков.** «Закрыто 5 из 12», «осталось 30 SP» — пересчитываются
   из источника и должны совпасть.
4. **Плейсхолдеры на месте.** Неизвестное помечено `[нет данных]`/`[не оценено]`,
   а не заполнено правдоподобным.
5. **Несекретный контент** (заголовки, статусы) сверяется в открытую на верность.

Не прошёл — **не показываем и не пишем**. Чиним вывод или явно помечаем пробел.

## Grounding-инварианты (дубль из SKILL.md, здесь — операционно)

- Источник любого факта — tool result, не память модели. Если не можешь указать,
  из какого вызова взят факт, — факта нет.
- Неизвестное → плейсхолдер. Догадка, поданная как факт, — провал gate-метрики.
- Re-hydration токенов — **после** прохождения гейта, client-side.
- **«Сейчас» — тоже факт, а не память.** Возраст задачи, «вчера», «N дней без
  активности» считаются относительно точки отсчёта *now*. Она инъектируется как
  вход (системное время вызова/тула), не берётся «из головы» модели. Иначе
  сдвинутый clock даёт неверный stall и ложное «успеваем».

## Измерение свежести данных

Свежесть — часть доверия: вывод «успеваем/не успеваем» без поправки на протухшие
данные есть «уверенная ложь», опаснее отсутствия ответа (BRIEF.md).

- Считай возраст задачи из `updated`/changelog (контракт `getActivity`,
  [tracker-mapping.md](tracker-mapping.md)).
- Тот же activity-механизм даёт **stall** (ADR-003): `in progress` N дней без
  изменения статуса/комментария/активности → жёлтый сигнал застоя **и** маркер
  «статус под вопросом».
- Помечай протухшее в выводе явно. Не «чини» — измеряй и подсвечивай.
- `N` (порог stall) — конфигурируемый параметр. Дефолт уточнить замером, но не
  оставляй неопределённым: даже priority-1 дейли должен с чем-то стартовать —
  разумная отправная точка **2 рабочих дня**.
- **Считай в рабочих, а не календарных днях.** Задача, не тронутая в выходные, —
  не застой. Календарный порог зажжёт ложный stall на каждой задаче в понедельник
  и подорвёт ровно то доверие, ради которого скилл существует. Учитывай выходные
  (и, если известны, нерабочие дни команды) при расчёте возраста и порога.

## Тесты гейта (обязательные)

- **Ноль выдуманных фактов:** на наборе режимов — ни одного id/числа/связи, не
  трассируемого в источник.
- **Плейсхолдер вместо догадки:** при отсутствии поля — плейсхолдер, не выдумка.
- **Утечка PII** (см. [redaction-rules.md](redaction-rules.md)): в LLM-payload
  нет сырых секретов.
- **Цикл блокеров:** обход графа с циклом X→Y→X не зацикливается, цикл показан.
- **Cold-start velocity:** без закрытых спринтов план не подставляет выдуманную
  velocity, а помечает основу.
- **Round-trip re-hydration:** гейт работает на masked-паре *до* re-hydration,
  поэтому сам маппинг `token→real` вне его покрытия — а карточка, восстановленная
  на неверного человека («назначено не тому»), это ровно экзистенциальный провал
  брифа. Ассерт: `rehydrate(tokenize(x)) == x` на всех токенах вывода (включая
  `⟦USER:…⟧`); один токен — одно реальное значение, без коллизий между сущностями.
