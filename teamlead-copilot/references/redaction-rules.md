# Redaction rules — обратимая токенизация перед LLM

Периметр banking-sensitive: содержимое задач может быть чувствительным, поэтому
**сырой текст карточек запрещено отправлять в managed/внешний LLM-коннектор без
редакции**. Этот слой стоит между tracker-MCP и моделью (ADR-002).

Предпочтение — **custom tool loop + self-hosted MCP** над managed-коннекторами:
контроль над тем, что покидает периметр.

## Принцип: токенизация, не разрушение

Маскирование **обратимо**. Классифицированное поле → стабильный плейсхолдер;
карта `token→real` живёт client-side и в модель не уходит.

```
⟦PII:7a3⟧   ⟦SECRET:0f2⟧   ⟦ACCT:c19⟧
```

Зачем обратимо, а не «звёздочки»: eval-гейт должен сверить вывод с источником по
структуре, а после прохождения гейта токены нужно вернуть в реальные значения для
показа тимлиду. Разрушающая маскировка обе задачи ломает.

## Инварианты токенизации

1. **Детерминизм в рамках сессии:** один секрет → один токен. Иначе модель не
   свяжет «того же человека»/«ту же задачу» между карточками (важно в дейли, где
   агрегируется максимум содержимого). **Под параллельным map-этапом** (дейли по
   людям) карта токенов — общий ресурс: воркеры должны делить одну карту (или
   детерминированную функцию `real→token`, например стабильный хеш), иначе один
   человек получит разные токены у разных воркеров и reduce «по людям» не
   сольётся. Тот же класс гонки, что и `since`-курсор в [tracker-mapping.md](tracker-mapping.md).
2. **Карта токенов — session-scoped state.** Её **нельзя** логировать в transcript
   модели, в eval-промпты, в отладочный вывод.
3. **Re-hydration только после eval-гейта**, client-side, внутри периметра.
4. **Несекретное не токенизируется.** Заголовок «fix payment timeout» уходит
   открытым — гейт проверит его контентную верность как обычно. Токенизируем
   только классифицированные поля.

## Классы полей (стартовая классификация — уточнить с владельцем периметра)

| Класс | Примеры | Токен | Токенизировать? |
|---|---|---|---|
| PII | ФИО клиента, телефон, email, адрес в описании | `⟦PII:…⟧` | да |
| Секрет | ключи, пароли, токены доступа в тексте карточки | `⟦SECRET:…⟧` | да |
| Финданные | номера счетов/карт, суммы по конкретному клиенту | `⟦ACCT:…⟧` | да |
| Внутр. идентификаторы | id задачи, ключ проекта, спринт | — | нет (нужны гейту для provenance) |
| Технический контент | заголовки/описания без PII, статусы, метки | — | нет |

`assignee` (член команды) — пограничный: имя коллеги обычно не banking-PII, но
если периметр требует — токенизировать как `⟦USER:…⟧` детерминированно, чтобы
сводка «по людям» оставалась связной.

## Второй канал утечки: доставка сводки

Редакция закрывает канал tracker→LLM. Но дейли доставляется как scheduled-сводка
в мессенджер (Slack/Telegram), а re-hydration возвращает реальные PII *перед*
показом — значит у периметра есть **второй выход**, и он не покрыт правилами выше.

- Если канал доставки **внутри** периметра — re-hydrate как обычно.
- Если **снаружи** (типичный мессенджер) — наружу не должны уходить
  re-hydrated PII/секреты. Варианты: доставлять сводку в маскированном виде, либо
  ограничить состав scheduled-сводки нечувствительными полями (статусы, счётчики,
  имена команды, если они не классифицированы как PII), не вытаскивая клиентские
  данные.
- Сам триггер и транспорт доставки строятся вне скилла, но **политику egress для
  них задаёт этот слой** — delivery нельзя трактовать как «чужую задачу» в обход
  редакции-блокера.

## Остаточный риск и обязательный тест

Порядок слоёв (ADR-002) переносит риск с «гейт слепнет» на «redaction пропустил
PII». Поэтому **тест на утечку сырого PII — gate-тест, обязателен**:

- gate: ни одного факта PII/секрета не должно покинуть периметр немаскированным;
- фикстуры с PII в `description`/комментариях; ассерт — в исходящем в LLM payload
  нет сырых значений, только токены;
- особое внимание — Daily-режим (максимум агрегированного содержимого);
- **оба канала утечки:** ассерт не только на LLM-payload, но и на payload
  scheduled-доставки во внешний мессенджер (см. «Второй канал утечки» выше) —
  при канале вне периметра в нём не должно быть re-hydrated PII.
